    autoMode()

    #Java   #Vertical prototype   <code>   <Javadoc>

    Objective: develop a cleaning robot system for a mobile robot platform based on Use case “Auto” mode.
    An on/off switch, as well as “auto”, “pause” and “home” buttons, are available to start cleaning in auto mode while the robot is generating a map of the cleaning area, to pause it or send it back to the charging station, respectively. The current status including map updates shall be sent to the app at regular time intervals. The robot is presumed that it was equipped with all sensors and effectors necessary for securely moving over plain and rough floors and not falling down any steps, for dry and wet cleaning, and for navigation with a laser scanner.

    Use Case Description

    Use Case AutoMode
    Related Requirements AutoMode: When the 'Auto' button is pressed, the robot starts cleaning while generating a map and also updating its location in that map.
    Initiating Actor User
    Actor's goal To make the robot clean while generating a map and updating its location
    Participating Actors APP, Drive system, Navigation system
    Preconditions
    • The “Auto” button is pressed.
    • Battery level is higher than the minimum battery value.
    • The robot is authorized and connected to the APP via local WIFI.
    Postconditions The cleaning robot goes back to charging dock when it finishes or the battery is less than minimum setting.
    Flow of Events for Main Success Scenario
    → 1. USER opens the APP and presses "Auto".
    ← 2. Drive system checks the current battery.
    3. include:: CheckBattery(To update current battery status)
    ← 4. Navigation system signals to the Robot System to scan the surrounding area for generating a map.
    5. include:: GenerateMap(To generate a map via LiDAR scanning and send the data to the App)
    ← 6. Navigation system signals to Robot system the path.
    7. Drive system operates.
    ← 8. Navigation System signals to the APP the location of the robot at regular time intervals.
    9. include:: SendLocation (UC-11)
    ← 10. Drive system keeps checking the current battery.
    11. include:: CheckBattery (To send the app the current location at regular time intervals)
    Flow of Events for Extensions (Alternate Scenarios)
    ← 2a. If the bettery level is less than minimum value, Drive system signals to the APP to alert and stay at the charging dock.
    6a. Robot System detects obstacles and floor material while cleaning.
    ← The Navigation system signals to the Robot system when it detects obstacles(stairs, furniture) and updates the new path.
    ← The Drive system signals to the Robot system to detect the floor material through the amount of dirt and operate in wet/dry mode.

    Activity diagram

    Class diagram

    Observer pattern: Child Observers run various conditions depending on Subject.
    Decorator pattern: decorations such as obstacle, segment are added on the map when they are detected.
    Iterator(traverse inherited classes of Observer), Singleton(boundaries have only one instance) also used.

    scheduleCleaning()

    #Java   #Vertical prototype   #MVC   <code>   <Javadoc>

    Objective: Develop a cleaning robot app for mobile devices based on Use case "schedule cleaning".
    The app is used to control the cleaning activities such as "auto" mode and "region/room" mode. Alongside, it is possible to plan the cleaning with the specific cleaning mode, time and area. In "region" or "room" mode, the user shall be able to define regions or rooms on the map that offers autogenerated area segments. Presume that the robot has been registered with the authorized app.

    Use Case Description

    Use Case ScheduleCleaning
    Related Requirements DefineRoom: In the “room” mode, the user can define rooms segmented by walls
    DefineRegion: In the “region” mode, the APP shall grant the user to define areas as regions.
    Initiating Actor User
    Actor's goal To schedule the future cleaning jobs
    Participating Actors Robot system
    Preconditions
    • The defined rooms and regions are empty.
    • The robot is turned on and charged.
    • User is logged in to the APP.
    • The map is already generated.
    Postconditions The data has been successfully saved.
    Flow of Events for Main Success Scenario
    → 1. USER opens the APP and presses “schedule”.
    ← 2. APP signals to Robot System to send the latest scanned data.
    → 3. USER names the cleaning job on the APP.
    → 4. USER selects the time to clean on the APP.
    → 5. USER decides whether or not to repeat the cleaning job on the APP.
    → 6. USER selects the days to to repeat on the APP.
    → 7. USER views the map on the APP.
    8. include:: ViewMap(To view the generated map in the APP)
    → 9. USER selects room(s) to clean on the APP.
    10. include:: defineRoom(To define rooms for the future planning)
    ← 11. APP signals to the Robot System to save the data [and Robot operates according to schedule.]
    Flow of Events for Extensions (Alternate Scenarios)
    → 6a. USER select the date to clean on the APP.
    6b. USER selects non-valide time.(e.g- time in the past)
    ← APP (a) detects an error, (b) pops up the alert, and (c) reset the time.
    → 9a. USER selects regions to clean on the APP.

    Class diagram

    Model–view–controller, Singleton(boundaries have only one instance) used.

    multiculture museum

    #Branding   #Graphics

    For my branding class assignment(2014), I designed branding for Multiculture Museum located in Seoul, Korea. People could enjoy miniature landmarks, currency, and traditional costumes there. It provided visitors with a variety of multicultural experiences and the feeling of traveling all around the world.
    The regular hexagon is geometrically stable and easy to expand, as seen in nature, like a honeycomb. I thought it was similar to a globe where each country communicates and expands its culture. The CMYK colour model and connections between each vertex represent the diversity of cultures intricately intertwined.
